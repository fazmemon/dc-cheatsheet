2 · Card / silicon evolution

The law: each generation ~doubles bandwidth via a MIX of faster lanes AND more lanes. "Which line card" = which silicon gen + port config.

The 4 levers (start at the lane, end at the chip)

A lane = one SerDes; its speed = bits/symbol × baud. A port = several lanes bonded. The chip = all lanes across all ports.

  • 1. bits/symbol — NRZ (1 bit) → PAM4 (2 bits).
  • 2. baud rate climbs each gen.
    1+2 = per-lane ladder 25 → 50 → 100 → 200G per lane.
  • 3. lanes per port (bonded) — e.g. 800G port = 8×100G or 4×200G.
  • 4. more total lanes / higher radix on the die.

Faster lanes (1+2) + more lanes (3+4) together = the doubling. The lane-count half (3+4) is the one people forget.

Simpler memory hook — 3 knobs: Bandwidth = bits × baud × lanes. bits (PAM4) · baud (faster symbols) · lanes (how many — bond more per port + more on the die). You always remember bits + baud; the one you drop is LANES.

Glossary

  • SerDes = serializer/deserializer; turns parallel bits into a serial stream on a lane. Port = several lanes.
  • radix = ports one chip fans out. High-radix → fewer Clos tiers.
  • HBM = stacked DRAM beside the chip = where the deep buffer lives (off-die).
  • VOQ = per-egress queues held at ingress → kills head-of-line blocking.
  • cells = deep chip chops packet into fixed cells, sprays across fabric, reassembles at egress.
  • Ramon = Broadcom fabric chip; Jericho = ports, Ramon = switch between them = scalable chassis.

3 · Packet flow — two flows, not three

Forwarding logic is identical (parse → lookup → forward). Buffering differs.

Shallow (Tomahawk / Trident) = output-queued shared-memory

wire → ingress → parse+lookup (ACLs here) → store ONCE in on-chip shared buffer → enqueue pointer to EGRESS queue → scheduler → wire.
No cells. No VOQ. No fabric. Single die, lowest latency, cut-through, microburst-limited. Buffer at egress.

Deep (Jericho / Q5) = ingress-VOQ + cell fabric

wire → ingress → parse+lookup → segment into cells → INGRESS VOQ (deep HBM) → egress scheduler credit/grant → spray cells across fabric → egress reassembles → wire.
No HOL, deep, scales across a modular chassis. ~µs latency. Buffer at ingress.

Why fabric exists: a deep box is a modular chassis = many line cards/chips; the fabric (+cells) connects them. A shallow box is a single die — nothing to cross, so no fabric. Fabric = multiple chips to interconnect. (Distinction is architecture, not box size: Nexus 9500 is a shallow chassis.)
QoS vs architectural latency: a strict-priority queue fixes queuing latency (voice doesn't wait behind bulk — SP edge does this on deep boxes daily). It CANNOT fix the architectural floor every packet pays on a VOQ box: cell seg/reassembly + credit grant + off-chip HBM (+ jitter). So deep+QoS = fine for ms-tolerant voice; fails for ns/deterministic HFT/RoCE → those need shallow cut-through (tens of ns) or L1 (~5ns).

One-liner: asymmetric needs all VNIs on all leaves; symmetric needs only the L3VNI — that's why symmetric won.

Unicast walk A→B (same subnet)

  1. Type-2 pre-loaded B's MAC+IP to every leaf via BGP.
  2. A ARPs → local leaf answers from its table = ARP suppression (never floods). The "IP" in Type-2 is what enables this.
  3. Ingress leaf encaps: inner frame + VNI + outer IP (src leaf loopback → dst leaf loopback).
  4. Spine sees outer IP only — blind to tenant MAC/VNI. Picks ONE spine via ECMP.
  5. Egress leaf reads VNI → decaps → delivers.

BUM (broadcast/unknown/multicast)

Type-3 (IMET) = the per-VNI flood list (who). Ingress replication = the how (leaf unicasts one copy to each leaf on the list; no multicast underlay needed = DC default). What's left after ARP suppression: real broadcast (DHCP) + multicast, not unknown-unicast.

Multihoming — ESI-LAG (= vPC, standards-based)

Say "DF election", not "broadcast suppression" (that's ARP suppression, a different thing).

Underlay / overlay

Underlay = IP reachability for VTEP loopbacks + ECMP. OSPF/IS-IS, or eBGP (RFC 7938) at hyperscale (per-leaf ASN, shared spine ASN, blast-radius contained). Overlay = BGP EVPN (spines = route reflectors). VXLAN needs jumbo MTU 9216 (~50B overhead) or silent blackhole.

5 · AI / GPU fabric

Scale-up vs scale-out

Scale-up = inside a node, GPUs glued by NVLink/NVSwitch (NVIDIA), ~TB/s. Scale-out = the network between nodes = what's sold: Ethernet (Juniper/Broadcom/Spectrum-X + UEC) vs InfiniBand (NVIDIA Quantum).

Why ECMP breaks for all-reduce

All-reduce = a few huge elephant flows. ECMP hashes per-flow → pins each to one path → elephants collide while links idle. Barrier-synced → slowest path stalls every GPU.

Fix = packet spraying / adaptive routing (spread one flow across all paths: Broadcom DLB, Spectrum-X adaptive, UEC). Safe ONLY on lossless + reorder-tolerant: RoCEv2 over PFC + ECN + DCQCN. NIC reassembles; fabric must not drop. Plain TCP = disaster.
"AI has incast — why shallow buffers, not deep?" (likely probe) Because AI solves congestion at the transport with lossless RoCE (PFC/ECN/DCQCN), not in the box. Deep buffers add the latency AI hates — so AI is deliberately shallow + lossless. Deep buffer is for traffic you can't make lossless: WAN, DCI, peering, mixed enterprise (big RTT, speed mismatch, classic TCP).

Non-Clos topologies (sold into AI/HPC)

  • Rail-optimized Clos — each GPU NIC on its own rail/leaf; same-rank GPUs = 1 hop (NVIDIA default).
  • Dragonfly — groups fully meshed by global links; cuts optics; needs adaptive routing (HPE Slingshot).
  • Torus — wrap-around grid, no spine; cheap, multi-hop (Google TPU).
  • OCS — optical circuit switch rewires topology per job (Google Jupiter).

Every non-Clos topology kills simple ECMP → all need adaptive routing + lossless. UEC = the open standard replacing per-flow ECMP + IB lock-in.

Fat-tree = the non-blocking (full-bisection) Clos these sit on (HPC name for spine-leaf; the underlay VXLAN rides on). A rail-optimized fabric is a fat-tree tuned to GPU ranks.

Sizing — how many GPUs fit (rail-optimized, 1:1, 8-GPU servers)

Juniper's AI scale-out answer (don't fall for "Q5")

Q5 is NOT the AI answer. Q5 (QFX10000) is the older deep-buffer DC switch. For the AI back-end fabric Juniper does what everyone does — buys Broadcom Tomahawk 5/6 (QFX5240 / QFX5250). (confirm current model names — AI naming moves fast.)
  • Why no in-house Jericho3-AI clone: the AI fabric is built shallow + lossless (Tomahawk + RoCE/PFC/ECN), not deep-buffer. Jericho3-AI is Broadcom's deep-buffer AI angle; most AI fabrics go shallow TH + lossless instead.
  • AI leaf/spine → Broadcom TH5/TH6 (QFX5240/5250) — merchant, same as Arista.
  • Scale across AI data centres (the J4 role) → in-house Express (PTX).
  • Differentiation = Apstra + AI-Native (Mist/Marvis) + open Ethernet/UEC — NOT the chip.
Room line: "For the AI fabric the silicon's merchant Tomahawk — same as Arista — so we don't compete on the chip; we compete on automation, openness and the operational model." Honest + strong.

7 · Juniper context (don't get caught out)

Juniper is now part of HPE — HPE acquired Juniper (~$14B, closed 2025) after a DOJ antitrust fight. Juniper now sits in HPE Networking alongside Aruba; Rami Rahim leads it. Know this — an interviewer may frame everything in HPE terms.

AI-Native Networking = their flagship story

Two different "AI" pitches — don't confuse them:

  • AI FOR networking = Mist (cloud-native AIOps) + Marvis (virtual network assistant). This is Juniper's headline differentiator — running the network with AI. Came from the 2019 Mist acquisition.
  • Networking FOR AI = the QFX / PTX AI-fabric (Tomahawk/Express, RoCE, packet-spray) — the GPU back-end fabric. This is the silicon side of the sheet.

Apstra (your pre-sales lever — know what it does)

Intent-based DC fabric automation (2021 acquisition): you declare the fabric intent → it generates device configs, validates cabling, and does continuous validation (tells you when reality drifts from intent). Key selling point: it's multivendor — manages Cisco, Arista, SONiC too, not just Juniper. So "open, no lock-in" is real.

Lossless detail (one layer under the names)

  • PFC (Priority Flow Control) = per-priority PAUSE — stop a queue instead of dropping.
  • ECN = mark packets at congestion instead of dropping; receiver echoes it back.
  • DCQCN = the RoCEv2 congestion algorithm that combines ECN (rate-cut signal) + PFC (last-resort no-drop).

Discovery questions to ASK them (SEs lead by asking)

  • "What's the GPU/accelerator and the collective library — does that point us at RoCE or InfiniBand?"
  • "Greenfield AI build, or extending an existing EOS/Nexus estate?"
  • "What scale — rack, pod, or cross-DD — and what's the power/cooling envelope per rack?"

8 · Chassis / line cards / optics (pre-sales hardware)

What's in a deep-buffer modular chassis

  • Line cards — the ports + the forwarding ASIC (e.g. Jericho). Packets enter, get looked up, buffered in the ingress VOQ. The silicon lives here.
  • Fabric modules (SFMs) — the internal switch carrying cells between line cards. Broadcom's fabric chip = Ramon (pairs with Jericho).
  • Supervisors / Route Engines — the control plane (runs BGP/EVPN, programs the line cards). Redundant pair.
  • Power supplies, fans.

Silicon generation maps to the LINE CARD

Each line card is built around one ASIC generation (Jericho2 → J2c+ → J3). The chassis is a generation-agnostic backplane — you populate it with whatever-gen line cards you buy. "Arista 7800R3 vs R4" = same chassis, different-gen line cards.

Fabric data path

ingress line card: lookup → ingress VOQ → segment into cells → spray across fabric modules → egress line card reassembles. More fabric modules = more internal bandwidth + redundancy.

Ramon is Broadcom-only (goes with Jericho). In-house vendors build their own fabric chips: Juniper (Q5/Express), Cisco Silicon One, Nokia FP5. Concept is universal (line-card ASIC + separate fabric ASIC); the chip is merchant-vs-in-house.

Upgrade to 800G without a forklift

Swap the line cards for newer-gen silicon with 800G ports (e.g. 100G Jericho2 cards → 800G Jericho3 cards). Likely add/upgrade fabric modules too so the fabric can feed them. Chassis, backplane, power frame persist — that's the whole point of modular: buy the chassis once, refresh line cards over a decade. (Fixed shallow box = single die, 1RU, no line cards/fabric — you replace the whole box.)

Optics (big AI cost + power lever)

  • Form factors: SFP28 (25G) → QSFP28 (100G) → QSFP-DD / OSFP (400/800G = the AI optics).
  • LPO (Linear Pluggable Optics) — strips the DSP, ~50-60% less power. Arista 7060XE7 headline; AI cost lever.
  • CPO (co-packaged optics) — optics next to the ASIC; the next power step at huge scale.
  • DAC (direct-attach copper) — passive, in-rack GPU-to-leaf, cheapest + lowest power. AOC = active optical cable.
  • Grey optics (single wavelength) = inside the DC, short reach, cheap. Coloured / DWDM = DCI over distance (multiplex onto scarce fibre) — lives at the deep-buffer DCI edge.

Cooling

Air vs liquid. AI boxes (Arista 7060XE7, Juniper QFX5250) are liquid-cooled. Why: chip power (TH6 at 100T+) + dense 800G optics + GPUs pack too many watts into too little space — air physically can't move the heat away fast enough. Liquid carries far more heat per volume (direct-to-chip cold plate / immersion). Power density outran air. Net: optics + power + cooling = the AI-fabric cost conversation = where pre-sales money actually is.

Oversubscription

Set at the leaf uplink:downlink ratio (sum of fabric-facing vs host-facing bandwidth). 1:1 (non-blocking) for AI / trading; up to 3:1 for general compute. The spine is non-blocking by construction — oversubscription is a leaf decision, not a spine one. (Weak answer says "the spine" — wrong.)
